Problématique

50-200
CVs par offre
+4h
Lecture manuelle par offre
  • CVs reçus par email — pas de standardisation
  • Lecture manuelle, répétitive et subjective
  • Aucune standardisation du scoring entre recruteurs
  • Temps perdu sur une tâche que l'IA peut absorber
  • Risque de biais et d'erreurs d'évaluation
Conséquence : le RH passe des heures sur le tri au lieu de se concentrer sur les entretiens et la décision finale.

Solution — TalentMatch

Soumettre une offre + un CV → analyse structurée en quelques secondes

  • Matching score justifié de 0 à 100
  • Points forts, lacunes et compétences manquantes identifiés
  • Recommandation typée : convoquer / attente / rejeter
  • Assistant conversationnel avec mémoire pour approfondir
  • Traitement asynchrone via file d'attente — réponse HTTP immédiate
Résultat : le RH gagne des heures par offre et prend des décisions éclairées par une analyse IA standardisée.

Modèle Conceptuel de Données (MCD)

Entités principales : USER, OFFRE, CANDIDAT, ANALYSE, AGENT_CONVERSATION, AGENT_CONVERSATION_MESSAGE

Note : ANALYSE résout le Many-to-Many entre OFFRE et CANDIDAT. Un candidat est toujours lié à une offre via une analyse.

Modèle Logique de Données (MLD)

Colonnes JSON :

  • required_skills (offres)
  • competences_extraites
  • langues
  • points_forts
  • lacunes
  • competences_manquantes

ENUMs :

  • recommandation : convoquer / attente / rejeter
  • status : pending / completed / failed
  • role : user / assistant
Note : les tables agent_conversations et agent_conversation_messages sont gérées automatiquement par laravel/ai.

Pourquoi pas un simple if/else ?

  • Le matching nécessite une compréhension sémantique des compétences
  • Un if/else ne peut pas interpréter « 5 ans d'expérience Laravel » dans un contexte RH
  • Le LLM capture les nuances et produit une justification lisible
Cycle de vie : status: pending (dispatch) → AnalyseCvJob (queue) → status: completed

Démo — Agent & Tools

Tool signature :

class GetCandidateAnalysisTool
{
    public function handle(int $candidatId): Analyse
    {
        return Analyse::with(['candidat', 'offre'])
            ->whereHas('offre', fn($q) => $q->where('user_id', auth()->id()))
            ->where('candidat_id', $candidatId)
            ->firstOrFail();
    }
}
Sans tool — Hallucination

« Le candidat a 10 ans d'expérience en Python. » — ❌ Donnée fabriquée

Avec tool — Données réelles

Tool appelle la BDD → réponse basée sur l'analyse réelle ✅

Mémoire : deux questions sur le même candidat sans perdre le contexte

  • Question 1 : « Quels sont les points forts de ce candidat ? »
  • Question 2 : « Et par rapport à l'autre candidat ? » → l'agent sait de qui on parle
